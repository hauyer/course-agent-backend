# 1.1.0 实施计划

## 1.0.0 现状审计

### 后端结构

- FastAPI 路由位于 `app/routers/`，统一使用 `get_current_user` 取得 JWT 用户。
- SQLAlchemy 模型位于 `app/models/`，1.0.0 共 17 张表；应用导入 `app.main` 时直接执行 `Base.metadata.create_all()`，尚未使用版本化迁移。
- Agent 唯一 HTTP 入口是 `AgentInterface`，路由同时提供流式和非流式接口。数据库消息是跨重启会话上下文来源，LangGraph 内存检查点只用于单次运行。
- `AgentState` 和 Runnable `configurable` 已包含 `user_id`、`course_id` 和 `top_k`。
- 语义检索实现位于 `vector_service.semantic_search()`。普通搜索接口和旧的 `agent_service` 调用该函数；Concept Agent 工具直接查询 Chroma，尚未复用统一服务。
- 当前 `ChatMessage.citations` 实际保存 Agent trace，接口固定返回空 citations；历史页再按 `kind=agent_trace` 从 citations 中拆出执行过程。
- 部分 Agent 工具在缺少配置时退回 `user_id=1`，不符合权限失败原则。
- 用户级 LLM 配置通过 `llm_config_service` 的 ContextVar 注入 Agent，可供知识图谱后台任务复用。

### 计划结构

- `Task` 已有 `due_at`、`estimated_minutes`、`course_id`、`priority`、`source`、`status`。
- `StudyPlanTask` 只保存 `study_plan_id`、`task_id`、`planned_date`、`sequence_no`。
- `StudyPlan.course_id` 已可为空；单课程计划接口和 Agent 计划工具已经存在。
- 当前创建计划任务会在同一事务中创建 Task 和关联，但没有多课程关联、预览、容量算法、幂等键或重新生成版本保护。

### 前端与桌面端

- `App.tsx` 负责真实导航注册；聊天、课程、资料、计划等较大页面已放在 `src/features/` 并使用懒加载。
- `ChatPage` 支持 SSE、多轮会话、折叠 Agent 过程和 Markdown，但引用展示依赖后端当前为空的数据。
- `PlansPage` 已包含单课程计划、任务和 Agent 创建入口，将以 Tab 增量接入综合规划，不新建孤立页面。
- 所有请求集中在 `src/api.ts`；新增请求继续使用明确方法和 TypeScript 类型。
- Electron 通过 PyInstaller 后端和本地 8000 端口运行；版本来自 `package.json`。数据库启动尚未接入迁移。

### CI

- 前端执行 `npm ci` 和 `npm run build`。
- 后端仅执行 `compileall`，没有开发依赖、pytest 或数据库迁移验证。

- Agent 响应和历史消息返回结构化 citations，trace 保持独立。

## 实现策略

### 可验证引用

- 使用 ContextVar 保存请求级 `CitationCollector`，在路由 `finally` 中复位。
- 统一检索服务先调用现有 Chroma 语义检索，再从 MySQL 以 user、course、material、chunk 四重条件验证结果。
- 检索顺序决定 C1、C2；按 chunk 去重，截取正文，流式和非流式共用同一收集器。
- 模型只看到程序生成的 `[C1]` 上下文；程序过滤不存在的引用编号。

### 综合规划

- 纯函数排程以日期、截止时间、优先级、剩余工作量和当前进度为输入，稳定排序并逐块分配，绝不超过每日容量。
- 预览不写数据库；确认和重新生成使用单事务、行锁、版本号及请求幂等键。
- 已完成任务和手工任务在重新生成时保留。

### 知识图谱

- 后台任务显式接收 user/course/job ID，并创建自己的 `SessionLocal`。
- 按有限 chunk/字符批次调用现有用户 LLM，严格 Pydantic 校验 JSON、限制关系白名单并验证证据 chunk。
- 新版本只有完整成功后才原子切换 active；失败保留旧版本。
- 前端真实轮询任务状态，在终态或组件卸载时停止。

## 向后兼容

- 保留所有 1.0 路由、模型字段和页面入口。
- 单课程计划默认 `plan_type=single`，旧数据迁移后自动获得默认值。
- 历史 citations 中旧 trace 数据在读取时兼容识别，但新写入不再混用。
- 新增字段均提供安全默认值；新表通过外键关联现有用户、课程、资料、片段、计划和任务。
- 备份格式保持可读取 1.0 数据；1.1 扩展数据采用可选区段，避免旧备份导入失败。

## 安全风险与控制

- 并发引用串线：ContextVar + 请求结束复位 + 并发测试。
- 越权 ID：服务层重复按 user_id/course_id 验证，不信任 Chroma metadata 或模型输出。
- 重复计划/图谱任务：幂等键、唯一约束、行锁和 running 冲突检查。
- 提示注入：资料只作为不可信学习文本，模型不能执行其中命令；证据 ID 必须属于当前输入批次。
- 数据破坏：迁移只增量创建/添加；已有数据库先校验和备份；图谱版本成功前不切换 active。
- 密钥泄漏：后台运行只通过用户级加密配置读取，响应、trace 和日志不输出密钥或资料全文。

## 测试计划

- 单元测试：CitationCollector、引用去重/清理、确定性排程、图谱规范化和输出过滤。
- 服务测试：用户/课程隔离、流式与非流式一致、计划事务回滚/幂等/再生成、图谱版本切换和独立 Session。
- 回归测试：认证、课程、资料、任务、单课程计划、笔记、学习记录、会话历史和备份关键路径。
- 前端：TypeScript + Vite 生产构建；通过组件逻辑验证轮询终态和清理。
- 迁移：独立测试数据库从 1.0 baseline 写入样例数据后升级 1.1，并核对核心记录数量和内容。
- CI：Windows 前后端检查；MySQL 集成测试独立 job，使用与任何正式 URL 不同的数据库名。

## 回滚方案

1. 发布前保留 `v1.0.0-local-backup` 和 1.0 数据库备份。
2. 应用代码可切回 `main` 或从标签创建 `restore/v1.0.0` 分支，不重写 1.1 分支。
3. 代码回滚到 1.0 前，执行 1.1 migration downgrade 只删除新增表/字段；必须先导出 1.1 新功能数据。
4. 若迁移或图谱构建失败，不继续后续写入；恢复数据库备份并保留错误日志的 trace ID。
5. 禁止用 Git 强制重置或数据库 DROP 操作代替回滚。
